# Реранкер: полная документация

## Введение

Векторный поиск в RAG-пайплайне возвращает top-K результатов, ранжированных по cosine similarity между эмбеддингом запроса и эмбеддингами чанков. Однако cosine similarity не всегда коррелирует с реальной релевантностью для конкретного запроса: документ может быть семантически «близок», но не содержать ключевых терминов из вопроса.

Реранкер решает эту проблему двухэтапным подходом:

1. **Широкий поиск** — из Qdrant запрашивается больше кандидатов (по умолчанию 20 вместо 5).
2. **Фильтрация + переранжирование** — отсечение по минимальному порогу cosine similarity, пересчёт скора с учётом keyword overlap, сортировка и усечение до нужного количества результатов.

## Как работает алгоритм

### Шаг 1: Threshold filtering

Все результаты с cosine similarity ниже порога `ScoreThreshold` (по умолчанию 0.3) отсекаются. Если после фильтрации не осталось ни одного результата, реранкер возвращает `nil` — генератор будет работать без контекста.

### Шаг 2: Hybrid scoring

Для каждого прошедшего фильтр результата вычисляется итоговый скор по формуле:

```
finalScore = (1 - KeywordWeight) * cosineScore + KeywordWeight * keywordOverlap
```

Компоненты:

- **cosineScore** — оригинальный скор из Qdrant (cosine similarity между эмбеддингом запроса и чанка).
- **keywordOverlap** — доля уникальных слов запроса, найденных в тексте чанка. Значение от 0 до 1.
  - Слова короче 2 символов игнорируются.
  - Сравнение case-insensitive.
  - Разбиение по пробелам и знакам пунктуации.
  - Дубли слов в запросе учитываются один раз.
- **KeywordWeight** — вес keyword overlap в итоговом скоре (по умолчанию 0.3, т.е. 70% cosine + 30% keyword).

### Шаг 3: Сортировка

Результаты сортируются по `finalScore` по убыванию.

### Шаг 4: Truncate

Из отсортированного списка берётся первые `ReturnTopK` результатов (по умолчанию 5). Скор каждого результата обновляется на `finalScore`, чтобы downstream-компоненты видели пересчитанное значение.

### Числовой пример

Запрос: `"Как работает обучение с подкреплением?"`

Допустим, после tokenize запроса получились слова: `["как", "работает", "обучение", "подкреплением"]` (слово "с" < 2 символов — отброшено).

| Чанк | cosineScore | Слова запроса в тексте | keywordOverlap | finalScore (kw=0.3) |
|---|---|---|---|---|
| A: «Обучение с подкреплением (RL) — метод...» | 0.82 | 3/4 | 0.75 | 0.7×0.82 + 0.3×0.75 = **0.799** |
| B: «Методы оптимизации в ML...» | 0.85 | 1/4 | 0.25 | 0.7×0.85 + 0.3×0.25 = **0.670** |
| C: «Reinforcement learning позволяет агенту...» | 0.78 | 0/4 | 0.00 | 0.7×0.78 + 0.3×0.00 = **0.546** |

Результат: чанк A (cosine 0.82) обогнал чанк B (cosine 0.85), потому что содержит больше ключевых слов запроса. Чанк C потерял позиции из-за нулевого keyword overlap (текст на английском).

## Место в пайплайне

### Поток данных `ask`

```
question
    │
    ├─ [QUERY_REWRITE_ENABLED=true] ──► rewriter.Rewrite(question) ──► +1-2 варианта
    │
    ▼
queries[]
    │
    ▼ embed каждый вариант
vectors[]
    │
    ▼ search(FetchTopK=20) по rag_structural для каждого вектора
[][]SearchResult
    │
    ▼ mergeResults (дедупликация по ChunkID, max score)
[]SearchResult
    │
    ├─ [RERANKER_ENABLED=true] ──► reranker.Rerank(question, results)
    │                                  1. threshold filter (≥ 0.3)
    │                                  2. hybrid score
    │                                  3. sort + truncate(ReturnTopK=5)
    │
    ├─ [RERANKER_ENABLED=false] ──► results[:5]
    │
    ▼
top results
    │
    ▼ generator.Generate(question, results)
    │
    ▼
ответ + источники
```

### Поток данных `eval` (3-mode)

```
question
    │
    ├──► RAG path:      embed → search(TopK=5) → generate
    │
    ├──► Reranked path:  embed → search(FetchTopK=20) → rerank → generate
    │
    └──► Baseline path:  generateWithoutRAG
    │
    ▼
CountFactHits для каждого пути → winner → отчёт
```

## Быстрый старт

Минимальный пример включения:

```bash
RERANKER_ENABLED=true task ask -- "Что такое декораторы в Python?"
```

Или через task-shortcut с преднастроенными параметрами:

```bash
task ask-reranked -- "Что такое декораторы в Python?"
```

Проверка эффективности через eval (3-mode сравнение):

```bash
task eval-reranked
```

## Конфигурация

Все параметры задаются через переменные окружения:

| Переменная | По умолчанию | Описание | Рекомендации |
|---|---|---|---|
| `RERANKER_ENABLED` | `false` | Включить реранкинг | `true` для production |
| `RERANKER_FETCH_TOP_K` | `20` | Количество кандидатов из Qdrant | Увеличивать при большом корпусе; >50 замедляет поиск |
| `RERANKER_RETURN_TOP_K` | `5` | Количество результатов после реранкинга | Совпадает с `EVAL_TOP_K`; увеличивать если генератору нужно больше контекста |
| `RERANKER_SCORE_THRESHOLD` | `0.3` | Минимальный cosine similarity для прохождения фильтра | Понижать для коротких/абстрактных запросов; повышать при зашумлённых данных |
| `RERANKER_KEYWORD_WEIGHT` | `0.3` | Вес keyword overlap в итоговом скоре (0..1) | `0` = чистый cosine; `1` = чистый keyword overlap; `0.3` — баланс |

## Рецепты настройки

### Только cosine (без keyword overlap)

```bash
RERANKER_ENABLED=true
RERANKER_KEYWORD_WEIGHT=0
```

Полезно, когда документы на иностранном языке или запросы формулируются абстрактно (мало буквальных совпадений слов).

### Агрессивная фильтрация

```bash
RERANKER_ENABLED=true
RERANKER_SCORE_THRESHOLD=0.5
```

Отсекает слабые результаты. Риск: при узком корпусе можно потерять все результаты.

### Больше кандидатов

```bash
RERANKER_ENABLED=true
RERANKER_FETCH_TOP_K=30
```

Расширяет пул кандидатов для реранкинга. Полезно при большом корпусе, когда релевантные чанки могут не попасть в top-20.

### Акцент на keyword overlap

```bash
RERANKER_ENABLED=true
RERANKER_KEYWORD_WEIGHT=0.7
```

Приоритет буквальных совпадений. Хорошо работает для технических запросов с точной терминологией.

### Комбинация с query rewrite

```bash
RERANKER_ENABLED=true
QUERY_REWRITE_ENABLED=true
```

Максимальный recall: rewrite генерирует дополнительные формулировки запроса, merge объединяет результаты, rerank переранжирует весь набор. Увеличивает латентность (дополнительный вызов Claude API на rewrite).

## Взаимодействие с Query Rewrite

Оба модуля **опциональны и независимы** — каждый можно включить/выключить отдельно.

При совместном использовании порядок в пайплайне:

1. **Rewrite** — Claude API генерирует 1-2 перефразированных варианта запроса.
2. **Embed + Search** — каждый вариант (оригинал + перефразированные) эмбеддится и ищется отдельно (с `FetchTopK` если реранкер включён).
3. **Merge** — `mergeResults` дедуплицирует по `ChunkID`, оставляя max score при дублях.
4. **Rerank** — реранкер работает с объединённым набором, используя **оригинальный вопрос** (не перефразированные) для keyword overlap.

Ключевой момент: реранкер всегда использует оригинальный вопрос пользователя для вычисления keyword overlap, даже если результаты были получены по перефразированным запросам.

## Оценка эффективности (eval)

### 3-mode eval

При `RERANKER_ENABLED=true` команда `eval` запускает три пути для каждого вопроса:

| Путь | Поток | Цель |
|---|---|---|
| **RAG** | embed → search(TopK=5) → generate | Базовый RAG без реранкинга |
| **Reranked** | embed → search(FetchTopK=20) → rerank → generate | RAG с реранкингом |
| **Baseline** | generateWithoutRAG | Claude без контекста из базы |

### Как читать отчёт

Для каждого вопроса выводится:

- **Fact hits** — сколько ожидаемых фактов найдено в ответе (case-insensitive substring match). Формат: `RAG 3/5 | Reranked 4/5 | Baseline 2/5`.
- **Winner** — путь с наибольшим количеством fact hits.
- **Sources** — источники (файл, секция, скор) для RAG и Reranked путей отдельно.

Сводная таблица в конце:

- **Wins** — сколько вопросов выиграл каждый путь.
- **Avg coverage** — среднее покрытие фактов в процентах.
- **Duration** — общее время выполнения.

### Winner logic

При равном количестве fact hits приоритет:

1. **Reranked** (если включён и набрал >= RAG)
2. **RAG**
3. **Baseline**

Если все три набрали одинаково — **Tie**. Если все ответы пусты — **N/A**.

### JSON-вывод

При включённом реранкере в JSON-отчёт (`eval_results.json`) добавляются поля:

- `reranker_enabled: true` — на уровне отчёта
- `rag_reranked_wins` — количество побед Reranked
- `rag_reranked_avg_facts_pct` — средний % покрытия фактов

Для каждого вопроса:

- `rag_reranked_answer` — ответ по Reranked-пути
- `rag_reranked_fact_hits` — количество fact hits
- `rag_reranked_matched_facts` — список совпавших фактов
- `rag_reranked_sources` — источники с пересчитанными скорами

## Архитектурные решения

### Почему threshold + keyword overlap, а не cross-encoder?

- **Ollama не поддерживает cross-encoder API.** Cross-encoder модели (типа `ms-marco-MiniLM`) требуют специализированного inference endpoint, которого нет в Ollama.
- **LLM-based scoring — высокая латентность.** Оценивать каждый из 20 чанков через Claude API — это 20 дополнительных API-вызовов (~5-10 секунд каждый). Неприемлемо для интерактивного использования.
- **Keyword overlap эффективен для русскоязычных текстов.** Русский язык богат морфологией, но даже простое contains-сравнение (без лемматизации) даёт значимый сигнал: ключевые технические термины обычно совпадают буквально.

### Почему FetchTopK = 20 по умолчанию?

Компромисс между recall и скоростью. 20 кандидатов — достаточно для того, чтобы реранкер мог «вытянуть» релевантные чанки, пропущенные в top-5 cosine. При этом поиск по 20 кандидатам в Qdrant практически не отличается по скорости от поиска по 5.

### Почему ScoreThreshold = 0.3?

Эмпирически подобранный порог для `nomic-embed-text`. Результаты с cosine similarity < 0.3 на практике нерелевантны. Порог достаточно низкий, чтобы не отсекать пограничные результаты, но достаточно высокий, чтобы убрать шум.

## Edge cases

### Все результаты ниже порога

Если после threshold filtering не осталось ни одного результата, реранкер возвращает `nil`. Генератор получает пустой контекст и отвечает на вопрос без RAG-контекста (аналогично Baseline-пути).

### Пустой запрос

При пустом запросе tokenize возвращает пустой список слов → `keywordOverlap = 0`. Итоговый скор вычисляется как `(1 - KeywordWeight) * cosineScore` — чистый cosine с пониженным весом.

### Дедупликация при query rewrite

Когда query rewrite включён, разные варианты запроса могут вернуть одни и те же чанки. `mergeResults` дедуплицирует по `ChunkID`, оставляя вариант с максимальным score. После этого реранкер работает с уникальным набором.
